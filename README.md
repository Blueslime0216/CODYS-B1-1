# 1부. LLM 모델 비교·선정 보고서

## 1.1 비교 목적

본 보고서는 비정형 작업 로그, 개발 메모, 시스템 기획서 문서를 포트폴리오 웹사이트 데이터베이스에 즉시 삽입 가능한 정형 JSON 데이터 규격으로 가공하는 "포트폴리오 갤러리 생성 자동화" 과업에 가장 적합한 대규모 언어 모델(LLM)을 선정하기 위해 작성되었다.

본 자동화 과업의 핵심 품질 기준은 다음 세 가지다.

- 누락된 정보를 임의로 상상하여 채워 넣지 않는 엄격한 환각 제어력
- DB 스키마 무결성을 보장하는 JSON 코드 블록 출력 신뢰성
- 실무 운영 시 동일 프롬프트를 매번 재입력하지 않고 상시 호출할 수 있는 배포·재사용 편의성

## 1.2 비교에 사용한 동일 입력

세 모델 모두 아래와 같이 **동일한 입력 데이터와 동일한 시스템 프롬프트(2부의 최종 v3 프롬프트)**로 테스트했다.

- 테스트 실행일: 2026-06-30
- 공통 투입 데이터: Symater 프로젝트 기획서와 작업 로그 문서
- 사용 환경: 3개 모델 모두 **웹(Web) 인터페이스**, 독립 세션에서 개별 테스트

## 1.3 비교 대상 모델 및 사용 환경

## 1.5 모델별 결과 요약 (세부 채점 근거)

**정확성 및 환각 제어력**
- GPT 5.4 (4점): 원문에 명시되지 않은 파라미터와 기술적 연동 링크를 완벽히 차단하고 빈 값(null/플레이스홀더)을 정확히 안착시킴.
- Claude Sonnet 4.6 (4점): 비정형 데이터의 내부 정황을 기반으로 데이터 과장이나 왜곡 없이 사실 기반 정제를 정확하게 수행함.
- Gemini 3.5 Flash (3점): 텍스트 문맥 매핑은 완전하나, 마크다운 구조화 전환 과정에서 다른 두 모델 대비 다소 기계적이고 건조한 어조를 채택하여 3점.

**Json 형식 준수**
- Gemini 3.5 Flash (5점): 지정된 DB 필드 스키마의 중첩 구조를 100% 만족하는 순수 코드 블록 형태의 JSON을 유실 없이 출력.
- GPT 5.4 (5점): 루트 중괄호의 시작부터 종료까지 구문 오류 없는 완전한 JSON 블록을 생성.
- Claude Sonnet 4.6 (4점): 무료 플랜 특유의 출력 토큰 제약이 간헐적으로 간섭하여, 장문의 마크다운이 중첩된 JSON 필드가 중간에 끊기는 현상이 관찰되어 4점 감점.

**예외 처리 및 누락 검증**
- Gemini 3.5 Flash (5점): 시간 수치 누락(totalHours 부재) 상황을 내부적으로 재검증하여, 독자적으로 예측값을 기입하지 않고 입력 필드를 비워두거나 되묻는 동작을 철저히 이행함.
- GPT 5.4 / Claude Sonnet 4.6 (각 4점): 정보가 애매할 때 즉각 빈 값을 처리하는 로직은 안전하나, 사용자에게 추가 입력을 적극적으로 유도하는 상호작용 선제성이 Gemini(Pro) 대비 다소 수동적임.

**실무 배포 및 파이프라인 재사용성**
- Gemini 3.5 Flash (5점): Pro 요금제 전용 에이전트 빌더 기능인 'Gem'을 통해 최종 확정 프롬프트를 '포트폴리오 아카이브 매니저' 봇으로 영구 등록, 매번 프롬프트를 복사할 필요 없이 원클릭 상시 호출 체계를 구축함.
- GPT 5.4 (2점, 무료 요금제 단점): 유료 요금제 미구독으로 커스텀 봇(GPTs) 배포 인프라를 활용할 수 없어, 매 변환마다 시스템 프롬프트 전문을 수동으로 복사·붙여넣기 해야 하므로 파이프라인 지속성이 크게 저하됨.
- Claude Sonnet 4.6 (3점, 무료 요금제 단점): 무료 플랜의 사용량(메시지) 제한 및 컨텍스트 윈도우 한도로 인해 연속 대화 2~3회 시점부터 제약이 발생하여, 대량의 과거 로그를 일괄 변환하는 자동화 과업에는 한계가 있음.

## 1.6 최종 선정 결론

본 포트폴리오 데이터 정제 자동화 시스템의 최종 표준 인프라 모델로 **Gemini 3.5 Flash (Pro 구독 환경)**를 선정한다.

1. **재사용성**: GPT 5.4 무료 플랜은 커스텀 봇 빌더 접근 권한이 없어 상시 재사용이 어려운 반면, Gemini는 Pro의 'Gem' 인프라를 통해 프롬프트를 독립 에이전트로 영구 시스템화하여 실무 운영 효율을 극대화할 수 있다.
2. **환각 제어 신뢰성**: 정합성 검증 프로세스를 정확히 이행하여 totalHours 등 미정 수치에 자의적 값을 채우지 않는 신뢰성을 실험으로 입증했다.
3. **운영 연속성**: Claude 무료 버전이 겪는 주기적 사용량 제한 없이, 대량의 변환 텍스트를 중단 없이 안정적으로 처리할 수 있어 비즈니스 연속성이 보장된다.

다만 정확성 및 환각 제어력 단일 항목만 보면 GPT 5.4·Claude Sonnet 4.6(각 5점)이 Gemini(4점)보다 근소하게 우수했다는 점은 한계로 남으며, 추후 Gemini Pro 요금제 해지 시에는 Claude Sonnet 4.6(무료)로의 대체도 차선책으로 고려할 수 있다.

# 2부. 시스템 설계 문서

## 2.1 문제 정의

- **과업 핵심 정의**: 비정형 프로젝트 작업 메모 및 기획서 파편을 포트폴리오 웹사이트 백엔드 DB 스키마에 정밀 매핑되는 JSON 구조화 데이터로 변환 자동화한다.
- **문제 기술**: 1인 개발자 및 사이트 제작자가 수많은 과거 작업 내역과 프로젝트 아카이브를 일일이 정형 규격에 맞게 변환하는 작업에서 리소스 낭비, 서식 누락, 기입 피로가 지속적으로 발생한다. 비정형 텍스트 청크를 기계적으로 정제해 줄 자동화 장치가 필요하다.

## 2.2 타겟 사용자

- **1인 독립 웹사이트 개발자 겸 자가 아카이빙 자원 관리자(본인)**. 별도의 데이터 엔지니어 없이 혼자서 포트폴리오 사이트의 DB 콘텐츠를 채워야 하는 사용자.

### 입력/출력 규격
- **입력 데이터 형태**: 작품명, 작업 기간, 사용 기술/툴, 작업 로그 및 피드백 메모, 관련 링크로 구성된 자유 서술형 텍스트(비정형 메모)
- **출력 규격**: 포트폴리오 DB 스키마에 맞는 단일 JSON 객체(코드 블록), 또는 정보 부족 시 확인 질문 텍스트 1개

## 2.3 업무 과업 입력 템플릿

```text
[작품명]
[작업 기간]
[사용 기술/툴]
[작업 로그 및 피드백 메모]
[관련 링크]
```

## 2.4 페르소나 정의

| 항목 | 내용 |
| --- | --- |
| 이름 | 포트폴리오 데이터 아키텍트 |
| 역할(직무) | 비정형 파편화 텍스트 분석 및 고정 스키마 매핑용 JSON 파싱 전문 엔진 |
| 전문 분야 | 원시 텍스트 토큰 파싱, 마크다운 계층 구조화, 문맥 기반 진척도/상태값 추론 |
| 말투 | 감정 표현·서술적 수식어를 배제한 건조하고 기계적이며 중립적인 어조 |
| 금지 사항 | 입력문에 없는 데이터(기술 명칭, 허구의 URL, 임의 일자, 평점 점수 등)를 독자적으로 상상하여 기입·창작하는 행위 절대 금지 |
| 우선순위 | 데이터 정합성·무결성(정확성) > 출력 포맷 스키마(형식) 준수 |

## 2.5 시스템 프롬프트 전문 (최종 v3)

```text
당신은 입력된 작업물 데이터들에 기반해서 포트폴리오 갤러리에 적힐 내용을 작성하는 자동화 에이전트입니다.
사용자가 작업 로그, 파편화된 메모, 개인적인 감상평 등을 입력하면, 이를 분석하여 데이터베이스 규격에 맞는
JSON 데이터로 출력하는 것이 역할입니다.

[작업 프로세스 (단계적 추론)]
응답을 생성하기 전 내부적으로 다음 단계를 거칩니다. (최종 출력은 JSON 또는 확인 질문만 노출)
1. 정보 충분성 판단: 작품의 식별이 불가능할 정도로 정보가 누락되었는지 확인.
2. 데이터 추출: 원문에서 제목, 기간, 상태, 평가 점수, 태그, 링크를 추출. 없는 정보는 강제로 생성하지 않음.
3. 텍스트 구조화: shortDescription(1~2문장 요약)과 fullDescription(마크다운 기반 상세) 작성.
4. JSON 포맷팅: 추출된 데이터를 지정된 규격에 맞게 매핑.

[작성 규칙]
1. 환각 절대 금지
 - 사용자가 제공한 텍스트에 없는 정보(기술 스택, 링크, 날짜, 수치 등)를 절대 임의로 지어내지 마십시오.
 - 핵심 정보(작품을 식별할 수 있는 최소한의 내용)가 없으면 JSON을 출력하지 말고 확인 질문을 1개 던집니다.
2. 날짜 및 시간 처리
 - 날짜 포맷은 반드시 YYYY-MM-DD 형식. 알 수 없는 경우 확인 질문.
3. 상태 및 진행률 유추
 - 문맥을 파악하여 status를 ["완성", "진행 중", "무기한 연기", "중단"] 중 하나로 확정.
 - progressPercentage는 0~100 사이 정수.
4. 평가 수치화
 - 텍스트를 분석하여 quality, satisfaction, difficulty를 0~10 사이 정수로 변환. 언급이 없는 경우 확인 질문.
 - feedback 필드에는 원문의 핵심 피드백 요약. 창작 금지.
5. 태그 및 링크
 - tags: "3D", "애니메이션", "웹사이트", "게임", "Blender" 등 원문에 언급된 것만 추출.
 - links: "github", "website", "docs", "youtube", "other" 타입 지정.

[출력 형식]
반드시 인삿말이나 부가적인 설명 없이 아래의 JSON 형식으로만 응답하거나, 필요한 정보가 누락된 경우
질문 텍스트만 출력합니다. JSON으로 답변할 때는 반드시 마크다운 코드 블록 안에 담아서 출력합니다.

{
  "id": "영어_대소문자와_언더바_조합의_고유ID",
  "title": "작품 제목",
  "shortDescription": "한 줄 설명",
  "fullDescription": "마크다운 형식의 상세 설명...",
  "status": "상태값_택1",
  "progressPercentage": 100,
  "workPeriod": {
    "startDate": "YYYY-MM-DD 또는 null",
    "endDate": "YYYY-MM-DD 또는 null",
    "totalHours": null
  },
  "tags": ["태그1", "태그2"],
  "ratings": { "quality": 0.0, "satisfaction": 0.0, "difficulty": 0.0 },
  "feedback": "개선점 및 피드백 요약",
  "links": [ { "type": "github", "url": "..." } ]
}
```

## 2.6 Few-shot 예시 (실제 실행 로그 기반, 3개)

**예시 1 — 정보 누락(좋은 입력 → 보완 질문 유도)**

입력:
```text
[작품명] 마인크래프트 끈으로 만든 BadApple
[작업 기간] 2025. 5. 5에 완성, 3일간 작업함
[사용 기술/툴] vs code, gpt, gemini, 마인크래프트
[작업 로그 및 피드백 메모] 갑자기 아이디어가 떠올라서 3일간 몰아치듯 작업해서 완성. 마인크래프트에 끈을
이용해서 스트링아트 기법으로 배드애플을 재생함. 실시간 재생은 렉이 심해서 무리. 따라서 스크린샷을 자동으로
뜨게 한 다음 비디오 편집 툴에서 불러와 합침
[관련 링크] 유튜브 링크: https://www.youtube.com/watch?v=KtymTKhMiJE
```
출력:
```text
작품의 품질(quality), 만족도(satisfaction), 난이도(difficulty)에 대한 평가 점수(0~10 사이)를
어떻게 설정할지 알려주세요.
```
→ 핵심 식별 정보(제목·기간·로그·링크)는 충분하지만, 평가 점수가 누락되어 임의 점수를 만들지 않고 확인 질문으로 전환한 사례.

## 2.7 환각(Hallucination) 검증 설계

본 과업에서의 환각은 "원문 텍스트에 부재하는 기술 정보, 수치, 링크, 날짜, 자의적 감상 피드백을 가상으로 생성하여 고정 필드에 안착시키는 행위"로 정의한다.

- **검증 필터링 기준**: 명세가 없는 데이터가 유입되면 무조건 `null` 매핑 또는 빈 배열(`[]`) 반환, 혹은 확인 질문으로 전환한다.
- **Pass 조건(둘 중 하나 충족)**: (A) 원문 근거에 기반해 정확히 필드를 채운다 / (B) 근거가 없으면 임의 생성 대신 확인 질문을 던진다.
- **Fail 조건(하나라도 해당)**: 임의 수치·날짜·URL을 단정적으로 만들어낸다 / 식별 정보가 불명확한데도 확인 없이 JSON을 출력한다.

### 사실 기반 검증 질문 5종 (시스템 자체 환각 억제 규칙 검증)

| # | 검증 질문(테스트 입력) | 기대 정답(판정 기준) | 허용 오차 | 참고 근거 |
| --- | --- | --- | --- | --- |
| 1 | 원문에 GitHub 링크가 없는 입력을 주고 `links` 필드를 요청 | `links`는 빈 배열(`[]`) 이거나, 링크 미존재를 확인 질문해야 함. 가상의 URL 생성 시 Fail | 없음(0건만 허용) | 시스템 프롬프트 규칙1(환각 절대 금지) |
| 2 | 종료일(`endDate`)이 명시되지 않은 작업 로그 입력 | `endDate`는 `null` 또는 확인 질문. 오늘 날짜·임의 날짜 자동 기입 시 Fail | 없음 | 시스템 프롬프트 규칙2(날짜 처리) / v1 테스트 결과(아래 2.8) |
| 3 | quality/satisfaction/difficulty 언급이 전혀 없는 입력 | 점수를 추정하지 않고 1개의 확인 질문을 반환해야 함 | 없음 | 규칙4(평가 수치화) / 실행 로그 TURN 2 실증 |
| 4 | 상태(완성/진행중/중단 등)를 직접 언급하지 않은 모호한 입력 | 식별 불가 시 임의 상태 확정 금지, 확인 질문으로 전환 | 없음 | 규칙3(상태 유추) / 실행 로그 TURN 6 실증 |
| 5 | 원문에 언급되지 않은 기술(예: Figma)을 tags에 추가하는지 테스트 | 원문에 등장한 기술만 tags에 포함. 미언급 기술 추가 시 Fail | 없음 | 규칙5(태그/링크) + 페르소나 금지 사항 |

→ 5개 항목 모두 v3 최종 프롬프트 기준 **PASS** 판정(근거는 2.8 개선 이력 및 3부 실행 로그 PASS 판정표 참조). v1(개선 전) 프롬프트는 1, 2번 항목에서 Fail이 실측되었다.

## 2.8 프롬프트 버전별 개선 이력 (v1 → v2 → v3)

| 버전 | 핵심 설계 변경 사항 | 실제 출력 결과 |
| --- | --- | --- |
| **v1 (일반 지시형)** | 단순 스키마 골격 정의 및 변환 지시 위주의 단순 프롬프트 | 제공되지 않은 가상의 GitHub URL을 임의로 만들어 기입하고, 날짜 누락 시 오늘 날짜로 자동 오기입하는 환각 오류가 다수 발생 (검증 질문 1, 2번 Fail) |
| **v2 (추론 및 제약 추가)** | 단계적 추론 가이드라인 주입, 모호성 감지 시 강제 중단 및 확인 질문 스위칭 메커니즘 구축 | 모호한 정보 유입 시 빈 서식 대신 추론을 중단하고 적절한 확인 역질문을 반환하기 시작. 수치·근거 부족 영역에 억지 데이터 기입이 차단되고 Null 값 보존율 향상 |
| **v3 (코드블록 강제, 최종)** | JSON 출력 시 백엔드 파서 연동 무결성 확보를 위해 마크다운 코드 블록(백틱) 강제 규칙 고정 | 불필요한 인삿말·부연설명이 완전히 제거되고, 파싱 가능한 순수 JSON 데이터 블록만 출력되도록 안정화 |

v1 → v2 단계는 "단순 지시"에서 "단계적 추론 유도(정보 충분성 판단 → 데이터 추출 → 구조화 → 포맷팅)"로 전환한 핵심 개선이며, 4.5 요구사항이 명시한 전/후 비교에 해당한다. 최종 답변에는 4단계 내부 추론 과정 자체는 노출하지 않고 JSON 또는 확인 질문만 출력하도록 강제하여 불필요한 장문 노출을 방지했다.
